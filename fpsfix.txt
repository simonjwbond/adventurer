# Performance and Rendering Optimization Strategy: "Adventurer"

This document outlines the specific technical strategies to fix the "dog slow" performance and color issues encountered in the Windows cross-compiled build.

## 1. Eliminate Draw Call Overhead: Render-to-Texture (RTT)
The current `TileMap::render` function performs 112 `SDL_RenderCopy` calls per frame. This is extremely inefficient on Windows where driver call overhead is higher.

**Strategy:** Create a static background texture once during initialization.
- **Why:** Reduces draw calls from 112 per frame to 1 per frame.
- **Implementation:**
  1. Create a "target texture" using `SDL_CreateTexture` with `SDL_TEXTUREACCESS_TARGET`.
  2. Set the renderer to draw onto this texture: `SDL_SetRenderTarget(renderer, backgroundTexture)`.
  3. Draw all 112 tiles once.
  4. Reset the render target: `SDL_SetRenderTarget(renderer, NULL)`.
  5. In the main loop, simply call `SDL_RenderCopy(renderer, backgroundTexture, NULL, NULL)`.

## 2. Fix the "Shades of Red" (Pixel Format & Endianness)
The red tint is caused by a mismatch between the manual pixel buffer (RGBA) and how the Windows Direct3D renderer interprets the `SDL_PIXELFORMAT`.

**Strategy:** Use `SDL_PIXELFORMAT_ABGR8888` for manual buffer filling on Little-Endian (Windows).
- **Why:** `SDL_PIXELFORMAT_RGBA32` is an alias that changes meaning based on endianness. On Windows, your manual R-G-B-A byte array matches the memory layout of `ABGR8888`.
- **Code Example:**
```cpp
// Explicitly define the format that matches your byte order in memory
// [Byte 0: R, Byte 1: G, Byte 2: B, Byte 3: A] = SDL_PIXELFORMAT_ABGR8888 on Little-Endian
SDL_Surface* surface = SDL_CreateRGBSurfaceWithFormat(0, width, height, 32, SDL_PIXELFORMAT_ABGR8888);
```

## 3. Configuration & Compilation Optimizations
The build is currently running in "Debug" mode (no optimizations) with high-overhead logging.

**Strategy A: Release Build (MANDATORY)**
- Use `-DCMAKE_BUILD_TYPE=Release` in the CMake command.
- This enables `-O3` optimizations, which are critical for the procedural loops in `ProceduralArt.cpp`.

**Strategy B: Remove I/O Bottlenecks**
- Remove all `fprintf(stderr, ...)` and `printf` calls from `Game::render`, `TileMap::render`, and `ProceduralArt::createTextureFromPixels`.
- Console I/O on Windows is synchronous and can block the main thread for several milliseconds per call.

## 4. Sub-Pixel Precision & Delta Time
The player movement currently feels "jittery" because it uses integer clamping.

**Strategy:** Accumulate movement in `float` but only round to `int` during the final `SDL_Rect` assignment in `render()`.
- **Why:** Prevents "stuttering" when moving diagonally or at speeds that aren't clean integers.

## 5. Summary of Specific File Changes

### graphics/ProceduralArt.cpp
- Switch to `SDL_PIXELFORMAT_ABGR8888` (for Little-Endian cross-compilation).
- Remove all `fprintf` calls inside loops.

### world/TileMap.cpp
- Add a `staticBackgroundTexture` member.
- Add a `generateStaticBackground(SDL_Renderer* renderer)` method.
- Update `render()` to use a single `SDL_RenderCopy`.

### CMakeLists.txt
- Add `set(CMAKE_BUILD_TYPE Release CACHE STRING "Build type")` to default to optimized builds.
- Add `-O3` to `add_compile_options` for GNU compilers.
